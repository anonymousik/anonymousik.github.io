[05]EXECUTION PIPELINE

Sequence of operations executed on every invocation.

1
Pre-Flight System Check
9 automated checks: Termux detection, Android API level, ~/storage access, network reachability via curl, disk space, dpkg lock removal, apt mirror validation, HOME permissions, scaffolding of ~/tools and ~/bin.
2
Base System Update
pkg update && pkg upgrade with live output. Core packages: git, curl, wget, zsh, tmux, openssh, man, procps, diffutils.
3
Module Installation
Each module runs sequentially with sf_run (critical, logs errors) and sf_try (optional, soft-fail). Animated spinner for git clones and rootfs downloads.
4
Shell Environment Configuration
Oh-My-Zsh + Powerlevel10k + 3 plugins (autosuggestions, syntax-highlighting, history-substring-search). Existing .zshrc backed up with epoch timestamp. MOTD injected as shell function.
5
Finalization
SSH ed25519 key generation (idempotent). Manifest written to ~/.secferro.conf. Full install log at ~/.secferro_install.log. Summary with context-aware next steps.

[09]FAQ
Is the script idempotent — safe to run multiple times?▾
Yes. Every operation checks state before executing: Oh-My-Zsh installs only when ~/.oh-my-zsh is absent, SSH key only when ~/.ssh/id_ed25519 is absent, Powerlevel10k and plugins only when their directories don't exist. Running the script again adds nothing if already complete.
Why was set -euo pipefail removed?▾
Android's restricted execution environment causes set -e to trigger false aborts on utilities like getprop (returns non-zero outside shell context) and any command using PIPESTATUS. v5.0.0 uses sf_run and sf_try wrappers that handle error codes per-command without aborting the script.
Why curl instead of ping for connectivity checks?▾
Android blocks ICMP (ping) without CAP_NET_RAW, which requires root. Without root the ping call hangs indefinitely rather than returning an error, causing the script to silently freeze. curl --max-time 8 to packages.termux.dev is reliable in all Android environments.
Can the script be used in CI/CD pipelines and mass deployments?▾
Yes — the primary design intent includes automation. Use --dev for fully unattended mode (no prompts), --no-ui for headless environments without Zsh, --dry-run in pipeline test phases. The script writes its manifest to ~/.secferro.conf for state introspection.
How do I check what was installed?▾
Install manifest: cat ~/.secferro.conf — lists version, codename, each module flag, and total non-critical error count. Full timestamped log: cat ~/.secferro_install.log.
