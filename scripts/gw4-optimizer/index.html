Diagnoza

Dlaczego One UI 8.0 wymaga tych poprawek?

Aktualizacja do Android 16 przyniosła mechanizmy jądra zaprojektowane dla procesorów big.LITTLE z 4–8 rdzeniami. Exynos W920 posiada tylko dwa identyczne rdzenie Cortex-A55 — brak rdzeni "big" powoduje, że nowe algorytmy działają kontrproduktywnie.

🔵
AOD → GPU voltage ramp bug
Sterownik Mali-G68 nie podbija napięcia szyny GPU wystarczająco szybko przy przejściu z trybu AOD (1 Hz partial GPU) do trybu aktywnego (60 Hz). Pierwsze klatki po wybudzeniu renderowane z minimalną częstotliwością → lag 1–2 s. Samsung nie wydał patcha sterownika (stan: 03/2026).
Fix B
🔴
PELT miscalibration (2-core A55)
Android 16 PELT skalibrowany pod big.LITTLE 4–8C. Na W920 wartości up_rate_limit_us=500µs powodują thrashing częstotliwości: agresywne skoki do 1.18 GHz → thermal throttle → stutter animacji. Korekta: up=1000µs, down=10000µs, sched_latency=8ms.
Fix D
🟡
zRAM / swappiness conflict
Samsung One UI 8.0 ustawia vm.swappiness=100 — agresywna kompresja zRAM. Na 2-core A55 dekompresja blokuje oba rdzenie → freeze 0.5–1.5 s przy otwieraniu Spotify, galerii, map. Korekta: swappiness=60 + extra_free_kbytes.
Fix E
🟠
HWC driver bug (flickering)
Hardware Composer driver Mali-G68 w buildzie R870XXU1JYLYL6 zawiera buga powodującego flickering i frame drops. Wbrew intuicji, wymuszenie Force GPU composition (SF 1008 i32 1) jest zalecanym obejściem — omija wadliwy HWC.
Fix C
🟣
Vulkan memory leaks
Stare sterowniki Mali-G68 + Android 16 HWUI pipeline Vulkan = wycieki pamięci przy renderowaniu interfejsu. Wymuszenie backendu SkiaGL (OpenGL ES) eliminuje problem — dojrzałe, stabilne sterowniki.
Fix C
🔋
WAKE_LOCK drenaż baterii
Google Assistant i Play Store trzymają stałe blokady energii po aktualizacji. cmd appops set ... WAKE_LOCK ignore ogranicza WAKE_LOCK bez wyłączania aplikacji — oczekiwana poprawa czasu baterii: 15–30%.
Fix F
⚠
Parametry jądra nietrwałe. Zmiany swappiness i schedutil resetują się przy restarcie zegarka. Dla trwałości wymagana aplikacja automatyzacji z wyzwalaczem On Boot (np. Tasker). Skrypt informuje o tym w każdym miejscu.

FAQ

Często zadawane pytania

Nie. Wszystkie operacje korzystają wyłącznie z uprawnień ADB (adb shell). Skrypt nie modyfikuje partycji systemowych (/system, /vendor) i nie wymaga żadnych modyfikacji bootloadera.

Niektóre parametry jądra (/proc/sys/, /sys/devices/) mogą wymagać roota do zapisu. Skrypt wykrywa brak uprawnień i informuje o każdej nieudanej operacji zamiast przerywać działanie.

Android 16 wprowadza nowe mechanizmy zarządzania energią i CPU (PELT, MGLRU, schedutil) zaprojektowane dla smartfonów z procesorami big.LITTLE posiadającymi 4–8 rdzeni.

Exynos W920 posiada tylko 2 rdzenie Cortex-A55 — brak rdzeni "big" sprawia, że algorytm PELT kalibruje częstotliwości zbyt agresywnie, powodując thrashing. Dodatkowo Samsung utrzymał swappiness=100 z poprzednich wersji — na 2-core CPU kompresja zRAM blokuje oba rdzenie.

Do tego dochodzą dwa błędy sterownikowe: HWC driver bug (flickering) oraz GPU voltage ramp bug (lag po AOD), których Samsung jeszcze nie naprawił aktualizacją sterownika.

Częściowo. Ustawienia zmieniane przez settings put (animacje, AOD, blur, phantom procs) są trwałe — zapisywane w bazie danych Androida.

Parametry jądra (swappiness, schedutil, sched_latency_ns) wpisywane do /proc/sys/ i /sys/devices/ są nietrwałe — resetują się przy każdym restarcie. Dla trwałości wymagana aplikacja automatyzacji, np. Tasker z wyzwalaczem On Boot, która wykona komendy ADB po każdym uruchomieniu.

Kompilacja ART (pm compile -m speed-profile -a) jest operacją intensywnie korzystającą z CPU przez 3–8 minut. Przerwanie procesu w połowie (np. wyłączenie zegarka z powodu rozładowania) może spowodować uszkodzenie profilów kompilacji i wymagać pełnego resetu.

Skrypt sprawdza poziom baterii przez dumpsys battery | grep level przed uruchomieniem i blokuje operację z wyraźnym komunikatem jeśli poziom jest poniżej 20%.

Normalnie Android używa Hardware Composer (HWC) do składania warstw UI bez angażowania GPU — jest to bardziej energooszczędne. Jednak sterownik HWC w Mali-G68 w buildzie R870XXU1JYLYL6 zawiera buga powodującego flickering i frame drops.

Komenda service call SurfaceFlinger 1008 i32 1 wymusza renderowanie przez GPU, omijając wadliwy HWC. Na W920 koszt energetyczny jest niewielki — GPU i tak pracuje przy renderowaniu UI — natomiast eliminacja flickeringu jest znacząca.

W poprzednich wersjach skryptu (v3.0) ta opcja była błędnie oznaczona jako "nie zalecana". Analiza inżynierska z stycznia 2026 wykazała, że jest to zalecane obejście dla W920.

Opcja R wykonuje Master Clear — usuwa wszystkie dane użytkownika z zegarka i przywraca ustawienia fabryczne. Jest to operacja nieodwracalna.

Skrypt wymaga dwustopniowego potwierdzenia (wpisanie RESET i POTWIERDZAM) oraz odliczania 30 sekund z możliwością przerwania dowolnym klawiszem. Po wykonaniu skrypt wyświetla instrukcję konfiguracji i kończy działanie, ponieważ autoryzacja ADB wygasa po resecie.

  • Backupy skryptu pozostają na hoście (PC / Termux)
  • Po restarcie należy ponownie przejść przez konfigurację ADB
  • Po konfiguracji uruchom skrypt ponownie i zastosuj Pakiet Kompleksowy (Z)

Skrypt automatycznie tworzy backup przed każdą modyfikacją w pliku ~/.gw4_optimizer/backup_YYYYMMDD_HHMMSS.txt. Opcja 9 z menu wyświetla listę dostępnych backupów i przywraca wybrany.

Jeśli brak backupu (np. pierwsze uruchomienie innej wersji skryptu), opcja 9 przywraca wartości fabryczne One UI 8.0 dla wszystkich modyfikowanych ustawień.
